Niouzou · Proposition de fonctionnalité

Discuter d'un article avec l'IA, sans quitter le fil.

Depuis n'importe quel article, l'utilisateur ouvre une conversation dont le contexte de départ est le résumé (ou le texte) de l'article. Il peut demander des précisions, creuser un point, ou élargir le sujet — un fil de discussion classique, ancré sur l'article.

Quatre placements sont proposés ci-dessous. Ils sont interactifs : appuyez sur « Ouvrir le chat » sous chaque téléphone pour ressentir l'interaction. Choisissez-en un, je rédige ensuite la spec puis on attaque le dev.

Recommandé Option A

Bottom sheet glissant

Une feuille remonte du bas et couvre ~78 % de l'écran ; l'article reste visible, assombri, derrière. Un rappel compact de l'article coiffe le fil. C'est exactement le pattern déjà en place pour le détail de score (ScoreDebugSheet) — geste connu, aucun dépaysement.

Pour

  • Garde le contexte de l'article à l'écran
  • Pattern déjà maîtrisé dans l'app
  • Retour au fil instantané (swipe down)
  • Place confortable pour un vrai échange

Contre

  • Un peu moins de hauteur qu'un plein écran
  • Clavier mobile ⇒ sheet à gérer finement

Option B

Écran plein dédié

Le chat pousse une nouvelle route (/articles/:id/chat) en plein écran ; l'article est rappelé dans une carte compacte en haut, dépliable. Le maximum de place pour une conversation qui dure, façon messagerie.

Pour

  • Espace maximal, idéal longues discussions
  • Le clavier a toute la hauteur
  • Route dédiée = partageable / historisable

Contre

  • Sort du fil — rupture de contexte
  • L'article n'est plus visible (juste rappelé)
  • Nouvelle route à câbler dans le routeur

Option C

Panneau intégré à la slide

Le chat se déplie dans le flux de l'article, sous le résumé. Tout reste sur une seule surface, on scrolle du texte vers la discussion. Le plus « organique »… mais en tension avec le scroll-snap du fil et la barre d'action flottante.

Pour

  • Zéro rupture : article + chat au même endroit
  • Continuité de lecture naturelle

Contre

  • Entre en conflit avec le scroll-snap plein écran
  • Composer + clavier coincés sous la barre d'action
  • Étroit : peu de place pour un vrai fil
  • Le plus délicat à intégrer proprement

Option D

Pop-up centrée

Un dialogue centré, via le composant Modal existant. Parfait pour une question rapide, une réponse, on referme. Léger à livrer — mais l'espace vertical est contraint et le ton « transactionnel » colle mal à une vraie conversation qui s'étire.

Pour

  • Réutilise Modal.tsx tel quel
  • Le moins de code à écrire
  • Focalise sur « une question »

Contre

  • Peu de hauteur — fil long à l'étroit
  • Sur mobile + clavier, la modale est comprimée
  • Ressenti « boîte de dialogue », pas « discussion »

Ma recommandation

Option A — le bottom sheet

Il coche le plus de cases sans dette : il réutilise un geste que l'app connaît déjà (le sheet de score), garde l'article présent à l'esprit, laisse une vraie place au fil, et se referme d'un glissement pour reprendre le swipe. L'option B reste le meilleur choix si tu veux, à terme, des conversations longues et historisées — on peut d'ailleurs livrer A d'abord puis « promouvoir » un fil en plein écran plus tard.

04

Nouveau réglage : modèle de conversation

Le chat mérite son propre modèle OpenRouter, distinct de celui de l'enrichissement (résumés/mots-clés) : la conversation veut un modèle plus dialogue/raisonnement, l'enrichissement veut un modèle bon marché et rapide. On ajoute donc une ligne dans PATCH /admin/config (chat_model) et un sélecteur dans l'écran Admin, juste sous le modèle d'enrichissement, en réutilisant le même sélecteur GET /admin/models.

Réglage séparé

Nouveau champ chat_model dans app_settings, exposé par GET/PATCH /admin/config. Défaut : le même que l'enrichissement, ou openrouter/auto.

Nouvel endpoint chat

POST /articles/:id/chat — reçoit l'historique, injecte le résumé/texte de l'article en system prompt, relaie vers OpenRouter. Idéalement en streaming (SSE) pour l'effet « en train d'écrire ».

Si l'IA est absente

Pas de clé OpenRouter ⇒ le chat est une fonctionnalité IA : on masque le bouton « Discuter » (comme les pins & l'écran Mots-clés aujourd'hui).

05

Ce que couvrira la spec

Une fois le placement choisi, je rédige un Epic/Story dans docs/EPICS.md + les mises à jour docs, portant sur :

Backend

Endpoint chat + service (routeur mince → service), construction du system prompt (résumé ⊕ texte tronqué à enrichment_input_max_chars), garde-fous (article de l'utilisateur, longueur d'historique), log de coût OpenRouter réutilisé.

Config & docs

chat_model dans DATA_MODEL.md, API_SPEC.md, ARCHITECTURE.md (variables d'env), et le sélecteur Admin dans DESIGN_SYSTEM.md.

Frontend

Composant de la surface retenue, point d'entrée, client API typé, états (chargement / streaming / erreur / IA absente), et le BlobBackground requis sur tout écran plein le cas échéant.

Questions ouvertes
